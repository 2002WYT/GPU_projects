# 五种 GPU SpMV 算法性能分析报告

> 测试平台：NVIDIA V100  
> 矩阵规模：`n = 2,000,000`  
> 对比算法：CSR Scalar、CSR Vector、CSR Adaptive、PCSR、LightSpMV  
> 测试线程块：128、256、512

---

## 1. 测试模式说明

| 模式 | nnz | 平均每行非零元数 | 矩阵特点 |
|---:|---:|---:|---|
| 0 | 83,666,594 | 41.83 | 行长主要为 3、8、16、32、64、128，不包含超长行 |
| 1 | 109,737,249 | 54.87 | 在模式 0 基础上加入少量 1024、4096 等长行，用于测试不规则负载与长行处理能力 |

模式 1 的非零元数量比模式 0 增加约 **31.16%**。

---

## 2. 核心结论

### 2.1 模式 0：CSR Vector 与 CSR Adaptive 并列最快

模式 0 不含超长行，最佳结果如下：

| 排名 | 算法 | 最佳 block | avg_ms | GFLOP/s | 相对最佳 CSR Scalar 加速比 |
|---:|---|---:|---:|---:|---:|
| 1 | **CSR Vector** | 512 | **5.13** | **32.61** | **3.04×** |
| 1 | **CSR Adaptive** | 512 | **5.13** | **32.61** | **3.04×** |
| 3 | LightSpMV | 256 | 5.20 | 32.18 | 3.00× |
| 4 | PCSR | 128 | 7.71 | 21.72 | 2.02× |
| 5 | CSR Scalar | 128 | 15.61 | 10.72 | 1.00× |

CSR Vector、CSR Adaptive 和 LightSpMV 的性能非常接近，三者时间差不超过约 **1.4%**。在没有超长行的矩阵上，CSR Vector 已能很好地利用 warp 并行，因此 Adaptive 的优势不明显。

### 2.2 模式 1：CSR Adaptive 成为最快算法

加入长行后，最佳结果如下：

| 排名 | 算法 | 最佳 block | avg_ms | GFLOP/s | 相对最佳 CSR Scalar 加速比 |
|---:|---|---:|---:|---:|---:|
| 1 | **CSR Adaptive** | 512 | **7.60** | **28.86** | **2.00×** |
| 2 | CSR Vector | 512 | 7.86 | 27.92 | 1.94× |
| 3 | LightSpMV | 256 | 7.94 | 27.65 | 1.92× |
| 4 | PCSR | 512 | 11.85 | 18.52 | 1.28× |
| 5 | CSR Scalar | 128 | 15.22 | 14.42 | 1.00× |

CSR Adaptive 比 CSR Vector 快约 **3.31%**，比 LightSpMV 快约 **4.28%**。这说明加入长行后，Adaptive 的分块策略开始发挥作用，能够更有效地处理不均匀行长。

---

## 3. block size 对性能的影响

| 模式 | 算法 | 最佳 block | 最差/最佳时间差 | 参数敏感性 |
|---:|---|---:|---:|---|
| 0 | CSR Scalar | 128 | 10.25% | 中等 |
| 0 | CSR Vector | 512 | 0.78% | 很低 |
| 0 | CSR Adaptive | 512 | 23.98% | 高 |
| 0 | PCSR | 128/256/512 | 0.00% | 极低 |
| 0 | LightSpMV | 256/512 | 26.92% | 高 |
| 1 | CSR Scalar | 128 | 2.17% | 很低 |
| 1 | CSR Vector | 512 | 1.65% | 很低 |
| 1 | CSR Adaptive | 512 | 20.26% | 高 |
| 1 | PCSR | 512 | 0.34% | 极低 |
| 1 | LightSpMV | 256 | 11.59% | 中等 |

可以看出：

- **CSR Vector 最稳定**，block 从 128 改到 512，性能变化不足 2%。
- **PCSR 几乎不受外部 block 参数影响**，因为它的第二个 kernel 使用固定线程块配置，并且整体瓶颈主要来自额外中间数组流量与同步。
- **CSR Adaptive 对 block size 最敏感**，512 在两个模式中均表现最好。
- **LightSpMV 在 128 线程时明显较慢**，256 已能达到最佳或接近最佳性能；继续增加到 512 并不会带来稳定收益。
- **CSR Scalar 不适合盲目增大 block size**，模式 0 下从 128 增加到 512 后时间由 15.61 ms 上升到 17.21 ms。

## 4. DRAM、SM 与 Occupancy 分析

### 4.1 五种算法整体呈现明显的访存受限特征

最快的 CSR Vector 和 CSR Adaptive 在最佳配置下，DRAM 指标达到约 **90%～94%**，而 SM 指标仅约 **5%～8%**。这说明计算单元并未成为主要瓶颈，性能主要受稀疏数据读取和间接访问 `x[col_idx[j]]` 的影响。

因此，对这类 SpMV：

- DRAM 吞吐率通常比 SM 利用率更能解释性能；
- 单纯增加计算并行度不一定继续提速；
- 减少额外内存流量、改善访存合并和负载均衡更重要。

### 4.2 Occupancy 高不代表 kernel 一定快

典型例子：

| 模式 | 算法与配置 | avg_ms | Occupancy |
|---:|---|---:|---:|
| 0 | CSR Scalar，block=128 | 15.61 | 97.63% |
| 0 | CSR Vector，block=512 | 5.13 | 73.85% |
| 0 | LightSpMV，block=512 | 5.20 | 99.93% |
| 1 | LightSpMV，block=256 | 7.94 | 49.90% |
| 1 | LightSpMV，block=512 | 8.12 | 99.61% |

CSR Scalar 虽然 Occupancy 接近 98%，但仍是模式 0 中最慢的算法。LightSpMV 的 Occupancy 从约 50% 提升到接近 100% 后，模式 1 的运行时间反而从 7.94 ms 增加到 8.12 ms。

这说明：

> Occupancy 只是 SM 上可驻留 warp 数量的指标，不等同于实际计算效率、访存效率或负载均衡程度。

对于 SpMV，达到足以隐藏访存延迟的 Occupancy 后，继续提高 Occupancy 往往不会产生明显收益。

### 4.3 CSR Scalar：高 Occupancy、低 SM、低带宽

CSR Scalar 的每个线程负责一整行。虽然可以同时驻留大量线程，但同一个 warp 中不同线程的行长不同，容易出现分支发散和负载不均衡。同时，各线程独立访问不同的 `x[col]`，访存合并效果较差。

因此出现：

- Occupancy 很高；
- SM 指标很低；
- DRAM 指标明显低于 CSR Vector；
- 最终运行时间最长。

### 4.4 CSR Vector：最稳定的通用实现

CSR Vector 使用一个 warp 处理一行，warp 内线程共同加载非零元并规约。它在两个模式下均表现稳定：

- block size 变化对时间影响不足 2%；
- DRAM 指标约 89%～94%；
- 模式 0 中与 CSR Adaptive 并列第一；
- 模式 1 中仅比 CSR Adaptive 慢约 3.3%。

如果矩阵行长没有极端不均匀分布，CSR Vector 是当前测试中最稳妥的通用方案。

### 4.5 CSR Adaptive：不规则矩阵上的最佳方案

模式 0 中没有真正的超长行，因此 Adaptive 主要执行短行分组路径，其性能与 CSR Vector 接近。

模式 1 加入长行后，Adaptive 可以针对短行和长行使用不同处理策略。其最佳时间为 7.60 ms，是模式 1 中唯一明显超过 CSR Vector 的算法。

这说明当前测试矩阵已经有效触发了 Adaptive 的设计目标：

- 短行合并处理；
- 长行由较多线程协同处理；
- 减少极端长行造成的尾部效应。

### 4.6 PCSR：近似带宽最高，但不是最快

PCSR 的 `approx_GB/s` 达到约 334～394 GB/s，高于其他算法，但运行时间明显更长。

原因是 PCSR 将一次 SpMV 拆成两个 kernel：

1. 计算每个非零元的乘积并写入中间数组；
2. 读取中间数组并按行求和。

因此它会额外产生：

- 中间数组写入；
- 中间数组读取；
- 第二个 kernel 的 shared memory 使用；
- 多次同步；
- 两次 kernel launch。

所以 PCSR 的 `approx_GB/s` 采用了更大的算法字节数作为分子，不能直接理解为“它的有效 SpMV 性能最高”。评价最终性能仍应以完整 PCSR 的 `avg_ms` 和整体 GFLOP/s 为主。

> 注意：PCSR 的 DRAM、SM 和 Occupancy 最好分别列出 `kernel1` 和 `kernel2`。若表中只保留一个值，应注明它是哪个 kernel 的数据，或明确说明采用了按运行时间加权的统计方式。

### 4.7 LightSpMV：动态调度有效，但对配置较敏感

LightSpMV 的 SM 指标明显高于其他算法，最高达到约 16%，说明动态行调度和 warp 内并行使执行单元更加活跃。

不过它也存在两个现象：

- block=128 时 Occupancy 约 25%，性能明显较差；
- block=512 时 Occupancy 接近 100%，但性能没有继续提升。

这意味着 LightSpMV 的主要优势来自动态负载均衡，而不是单纯追求最高 Occupancy。当前矩阵上，block=256 是更稳妥的选择。

---

## 5. 模式变化带来的影响

从模式 0 切换到模式 1 后：

- nnz 增加约 31.16%；
- CSR Vector、CSR Adaptive、PCSR 和 LightSpMV 的最佳运行时间增加约 48%～54%；
- CSR Adaptive 的增长幅度相对最小；
- CSR Scalar 的运行时间基本没有增加，但 GFLOP/s 显著提高。

CSR Scalar 的这一现象说明模式 0 中大量短行使每个线程完成的工作过少，线程调度、行边界读取和负载不均衡占比较高。模式 1 增加长行后，部分线程拥有更多连续工作，单线程指令级并行和有效访存比例提高，因此即使 nnz 增多，总时间仍接近不变。

不过，这也意味着 CSR Scalar 的耗时不再近似与 nnz 成线性关系，后续最好再使用真实 Matrix Market 矩阵验证这一现象。

## 6. 推荐配置

| 使用场景 | 推荐算法 | 推荐 block |
|---|---|---:|
| 行长较规则、没有极端长行 | CSR Vector | 256 或 512 |
| 行长高度不均匀、包含长行 | CSR Adaptive | 512 |
| 希望得到稳定且简单的实现 | CSR Vector | 256 |
| 动态负载均衡实验 | LightSpMV | 256 |
| 研究两阶段 SpMV 与中间数组行为 | PCSR | 参数影响较小 |
| 仅作为基础对照 | CSR Scalar | 128 |

综合当前两种矩阵，建议将：

- **CSR Vector** 作为通用基线；
- **CSR Adaptive，block=512** 作为不规则矩阵的首选；
- **LightSpMV，block=256** 作为动态调度方案；
- **PCSR** 作为具有额外访存开销的对照算法；
- **CSR Scalar** 作为最基础的性能下界。

---

## 7. 最终实验结论

本实验表明，V100 上的 SpMV 主要受显存访问和不规则索引访问限制，而不是浮点计算能力限制。CSR Scalar 虽然具有很高的 Occupancy，但由于 warp 内负载不均衡及访存效率较低，性能最差。CSR Vector 能稳定地利用 warp 协作，在规则和中等不规则矩阵上均具有良好表现。CSR Adaptive 对 block size 较敏感，但在包含长行的模式 1 中取得最高性能，证明其根据行长选择不同执行策略的设计适合不规则稀疏矩阵。LightSpMV 通过动态调度提高了 SM 活跃程度，但其性能并不随 Occupancy 单调增加。PCSR 虽然展示出较高的近似数据吞吐率，但由于引入中间数组和双 kernel 执行，其整体运行时间仍明显高于 CSR Vector、CSR Adaptive 和 LightSpMV。

当前结果中，模式 0 的最佳性能为 **5.13 ms、32.61 GFLOP/s**，由 CSR Vector 和 CSR Adaptive 共同取得；模式 1 的最佳性能为 **7.60 ms、28.86 GFLOP/s**，由 CSR Adaptive 在 block=512 时取得。

## 9. 数据说明

- `avg_ms`：程序使用 CUDA Event 测得的一次 SpMV 平均执行时间。
- `GFLOP/s`：按照每个非零元一次乘法和一次加法，即 `2 × nnz` 计算。
- `approx_GB/s`：根据程序设定的数据访问模型计算的近似带宽，不等同于 NCU 的实际 DRAM 带宽。
- `DRAM %`：Nsight Compute 报告中的 DRAM 吞吐率相对峰值百分比。
- `SM %`：Nsight Compute 报告中的 SM 吞吐率相对峰值百分比。
- `Occupancy %`：实际活跃 warp 数相对于硬件最大活跃 warp 数的比例。
- PCSR 包含两个 kernel，其微架构指标原则上应分 kernel 报告。
